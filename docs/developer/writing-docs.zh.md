# 编写开发者文档

[English](writing-docs.en.md) | **[中文](writing-docs.zh.md)**

在本目录中创建和维护文档的指南。

## 目的

这些文档主要面向 AI 编码代理，其次面向人类。它们应该：

- 解释阅读代码时不明显的模式
- 引导 AI 代理避免常见错误（旧模式、反模式）
- 保持"常青" - 随应用演进更新

它们**不应**教授库（AI 了解 Zustand、React Query 等）或重复代码。

## 何时编写新文档

| 情况               | 操作              |
| ------------------ | ----------------- |
| 新的架构模式或系统 | 创建新文档        |
| 对现有系统的增强   | 更新现有文档      |
| 一次性实现细节     | 不记录 - 代码中有 |

新文档应添加到本目录的 `README.md` 中。

## 内容指南

### 包含

- 非显而易见决策背后的"为什么"
- 代码中不明显的模式的"如何添加 X"部分
- 常见错误（尤其是过时 AI 训练数据中的）
- 相关文档的交叉链接：`See [state-management.zh.md](./state-management.zh.md)`

### 排除

- 不影响使用的实现细节
- 库教程
- 检查清单（AI 会强迫性地生成这些 - 要抵制）
- "未来增强"或推测性部分
- 重复代码库的大型代码块

## 代码示例

保持示例最小化。使用 ✅/❌ 格式：

```typescript
// ❌ BAD: Subscribes to entire store
const { visible } = useUIStore()

// ✅ GOOD: Selector for specific value
const visible = useUIStore(state => state.visible)
```

有帮助时引用实际文件："See `src/lib/menu.ts` for the full implementation."

## 格式化

| 元素       | 用途                   |
| ---------- | ---------------------- |
| 表格       | 比较选项、快速参考     |
| ✅/❌ 示例 | 正确 vs 错误模式       |
| `行内代码` | 文件路径、命令、函数名 |
| 标题       | 可扫描的结构           |

保持文字简洁。直奔主题。

## Token 效率

不是主要目标，但是好习惯：

- 比较时用表格而非文字
- 不解释 AI 已知的内容
- 链接到其他文档而非重复内容
- 永远不要为简洁牺牲清晰度

## 维护文档

当你更改文档描述的模式时更新文档。如果不确定代码更改后文档是否需要更新，读一遍 - 你会知道的。
